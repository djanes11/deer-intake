# Pilot Support Playbook

Use this document to support pilot processors consistently.

## Support Expectations

- Acknowledge pilot issues quickly.
- Separate blockers from improvements.
- Never promise a same-day feature unless it is already scoped and safe.
- If there is uncertainty, give a workaround first and a follow-up time second.

## Severity Levels

### P1

The processor cannot intake deer, search deer, print required paperwork, or access staff login.

Response:

- treat as immediate
- provide workaround if possible
- communicate status until resolved

### P2

The shop can still work, but a key feature is degraded.

Examples:

- SMS down
- scan workflow unreliable
- print queue mismatch

Response:

- same-day triage
- workaround documented
- follow-up after fix

### P3

Usability pain, wording problems, or requested improvements that do not block operations.

Response:

- log clearly
- group by theme
- review weekly

## Pilot Triage Template

For every issue, capture:

- processor name
- time reported
- user role
- exact page/flow
- expected behavior
- actual behavior
- screenshot or steps if available
- workaround given
- final resolution

## Weekly Pilot Review

At least once per week, summarize:

- blockers resolved
- active risks
- repeated usability pain
- requested enhancements
- whether the processor still feels confident using the app

## Pilot Exit Criteria

A processor is ready to move from pilot to paid/steady use when:

- owner uses the system without hand-holding
- staff can complete core daily tasks
- at least one fallback path has been tested
- the processor wants to continue after seeing real deer move through the system
